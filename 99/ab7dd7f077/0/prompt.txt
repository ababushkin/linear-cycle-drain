If its small, then it can be a single shot piece of work that can be immediately merged into main: UNLESS it requires human review and won't block other jobs in the queue. The other thing it should do it to verify if the AC is good enough to meet all the behavioural changes being introduced.

Verifier is not the same thing a code-review - that's already been done during implementation via a sub-agent. Verifier should be strictly focused on correctness - did the implementor build what was described in the spec? Were the goals reached? Might need a better name for this role.

We need another role, which is Pre-Human-Review Preparer (or something better) - this role specifically goes over all the PRs (or Graphite PRs if that's what been authored) and makes sure that all the PRs have been written in the following format:

- What
- Why
- What the reviewer should focus on (i.e. anything complex/tricky/risky that the agent would like the human to pay attention to?) -- it should only be important/critical elements. If there's nothing then actually the work can be submitted for auto-merge.
