<title>Guard worktree teardown on the Done path so a removal failure doesn't crash the drain</title>
<description>
## Problem

On the Done path, `orchestrator.run()` calls `worktree.remove(target_repo, worktree_path)` outside any try/except. `worktree.remove` raises `RuntimeError` on a non-zero `git worktree remove` exit. So a teardown failure becomes an uncaught traceback **after** the issue is already recorded Done in Linear and written to the run log — the worst place to crash, because the drain dies mid-cycle with the bookkeeping already committed.

This is a pre-existing gap, surfaced during the code review of "Workers inherit the repo's project config inside worktrees". That change slightly raises the probability of a teardown failure: if an operator configures a `worktree_config_paths` entry that is neither gitignored nor tracked, the worker's `git add` stages the symlink and `git worktree remove` then refuses the dirty worktree (exit 128). The config precondition is now documented, but the unguarded teardown remains.

## Outcome

A teardown failure on the Done path is recorded (run-log note / warning on stderr) and the drain continues or halts cleanly, rather than crashing with a traceback after the issue is already Done.

## Acceptance criteria

1. Wrap the success-path `worktree.remove` so a `RuntimeError` is caught and converted to a recorded outcome (warning + run-log field, or a clean halt — pick per the existing halt contract), not an uncaught traceback.
2. Test covering a `worktree.remove` failure on the Done path that asserts the drain doesn't crash and the failure is recorded.

## Notes

Low priority / backlog — pre-existing behaviour, not a regression. Slot into a future cycle deliberately.
</description>
