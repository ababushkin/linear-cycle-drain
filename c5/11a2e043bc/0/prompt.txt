<issue identifier="ABA-234">
<title>drain-cycle: a live run is uninspectable — `active.json` marker + `status` command</title>
<description>
## Problem

A running `drain-cycle` is hard to observe. Mid-issue there is almost no signal: the run-log gains an entry only when an issue *finishes*, the orchestrator prints only three sparse stderr lines, and there's no way to inspect from a second terminal or when the run was launched detached. The operator cannot tell a working run from a hung/runaway one.

## Change

Builds on the streamed worker (<issue id="eb62c999-a798-4055-9d3c-7993542fbdf8">ABA-236</issue>), so the live view is **token-aware**:

1. **Active-issue marker** (`drain_cycle/progress.py`) — the orchestrator writes `~/.drain-cycle/active.json` just before each spawn and clears it as soon as the worker returns (try/finally over done/halt/timeout/breach). Lives **above** `runs/` so `grade`'s `runs/*.json` glob never sees it. Carries: `pid`, `cycle_id`, `run_log_path`, `issue{identifier,title,repo,worktree_path}`, `model`, `started_at`, `index`, `total`, and a live `progress` block updated on every stream event (`turns`, `cumulative_tokens`, `peak_context_tokens`, `cost_usd`, `elapsed_seconds`, `last_event_at`).
2. `drain-cycle status` (`drain_cycle/status.py`) — reads the marker from any terminal: which issue, repo, model, `[index/total]`, elapsed, turns, and **tokens (cumulative + peak) with cost shown together** against the configured caps (e.g. `12.4M / 8M ⚠`). Detects a stale marker (pid gone → crashed/killed) and a no-marker state (points at `grade`).
3. **Compact stderr progress line** per stream event for terminal-watching, e.g. `ABA-205 · turn 42 · 8.1M tok (peak 180k) · $12.30 · 14m`.

## Acceptance criteria

* `drain-cycle status` during a run shows the current issue, repo, model, elapsed, turns, cumulative + peak tokens, and cost — vs caps.
* `drain-cycle status` with no run prints a no-active-run message and exits 0.
* A stale marker (pid not alive) is reported as such, not as a live run.
* Marker written before spawn, removed after the worker returns on every exit path.
* `grade` output is unaffected by the marker file.
* New tests for `progress`, `status`, the orchestrator marker lifecycle, and the `status` CLI dispatch; full suite green.

## Recommended worker model

**Sonnet** — moderate, well-specified.

## Files

`drain_cycle/progress.py` (new), `drain_cycle/status.py` (new), `drain_cycle/orchestrator.py` (marker write/clear + index/total), `drain_cycle/cli.py` (`status` dispatch + usage), `tests/`, `README.md`, `docs/design-decisions.md` (marker-location decision).

Depends on the streamed worker (<issue id="eb62c999-a798-4055-9d3c-7993542fbdf8">ABA-236</issue>) for the live token feed.
</description>
