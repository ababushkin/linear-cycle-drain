## Problem

drain-cycle orders the issues it executes by `(priority, sortOrder)` (Linear priority first, manual drag-order only as a tiebreak) and **ignores blocks/blocked-by entirely** (`drain_cycle/linear.py` `_sort_pending_issues`). The operator maps execution via the manual drag-order *and* the blocks graph, so today is wrong on two counts:

1. Priority overrides the manual order ‚Äî an issue dragged to the top is still picked after every higher-priority issue.
2. A blocked issue can run before its blocker, wasting a worker on work that can't succeed.

Measured on cycle 3 (23 pending issues, 27 intra-cycle blocks edges): <issue id="20943999-2ee5-4a5b-a94a-b3cded031011">ABA-274</issue> **runs last (position 23)** though it blocks six issues (<issue id="35d48cd5-804e-4045-9f0f-d467be525963">ABA-275</issue>‚Ä¶280) that all run before it; <issue id="f6534b45-8158-4b9e-92d5-ba287c5a0a24">ABA-291</issue> **runs first** though it's blocked by <issue id="1980db92-316c-4d2e-817b-fa5b4669349d">ABA-265</issue> and <issue id="00f1876f-6e44-4b05-a65c-07455865d481">ABA-266</issue>. A blocks-aware, manual-order drain corrects both (<issue id="20943999-2ee5-4a5b-a94a-b3cded031011">ABA-274</issue> ‚Üí \~17, before all six; <issue id="f6534b45-8158-4b9e-92d5-ba287c5a0a24">ABA-291</issue> ‚Üí \~6, after both blockers).

## Decisions

1. Order purely by manual `sortOrder` ascending; **drop** `priority` **from ordering** (remove the `(priority, sortOrder)` key and the `_NO_PRIORITY_SORT_KEY` remap).
2. **Respect blocks:** topological order over the runnable set, `(sortOrder, id)` as the tiebreak among ready issues ‚Äî so manual order is honored as far as dependencies allow.
3. **External unresolved blocker ‚Üí defer.** If a pending issue is blocked by an issue not in this drain's pending set (current cycle Todo/Backlog) and that blocker is not `completed`/`canceled` ‚Äî this includes an issue already In Progress in the cycle, which this drain won't complete ‚Äî defer the blocked issue: leave it Todo, don't spawn, log to stderr. **Deferral cascades:** if X is deferred and X blocks Y within the drain, Y defers too. Deferrals are **not** run-logged ‚Äî a deferred issue was never attempted, so recording it would corrupt `grade`'s `done/attempted` completion signal.
4. **Blocks cycle ‚Üí halt.** If the blocks graph among pending issues contains a cycle (self-loop included), set `cycle_halt_reason`, run nothing, exit 1, name the issues.

## How ‚Äî data

In `linear.pending_issues` GraphQL: **remove** `priority`; **keep** `sortOrder`; **add** `inverseRelations { nodes { type issue { id identifier state { type } } } }`. For issue Y, `inverseRelations` filtered to `type == "blocks"` yields Y's blockers (the `issue` field) with their state ‚Äî sufficient for both intra-set edges and external-blocker detection; `relations` (forward) is not needed. Filter out non-blocking types (`related`, `duplicate`). In post-processing (beside the existing `labels` flatten), attach `issue["blockers"] = [{id, identifier, state_type} ‚Ä¶]` and drop the raw `inverseRelations` key ‚Äî wire-shape stays local to `pending_issues`, like `labels`.

## How ‚Äî algorithm

New pure function `_plan(issues) -> ExecutionPlan` replaces `_sort_pending_issues` (stays the directly-tested unit; reads `issue.get("blockers", [])` so stubs need no new field, mirroring `model.resolve`'s `issue.get("labels", [])`). Add `@dataclass(frozen=True) ExecutionPlan(order, deferred)` and `class DependencyCycleError(RuntimeError)` (carries the involved identifiers) in `linear.py`, imported by the orchestrator like `RepoResolutionError`. A **single Kahn pass** does ordering, cascade, and cycle detection:

1. `P` = pending issue ids. Build a **de-duped** intra-P edge set (blocker‚Üíblocked) from each issue's `blockers` where `blocker.id ‚àà P`, plus in-degree counts (set-based so duplicate relations and self-loops can't inflate in-degree).
2. Repeatedly take the ready node (in-degree 0) with the smallest `(sortOrder, id)`. Classify it: **deferred** if it has an external unresolved blocker (`blocker.id ‚àâ P` and `blocker.state_type` neither `completed` nor `canceled`) **or** any intra-P blocker was deferred (cascade falls out for free, since Kahn takes a node only after all its intra-P blockers); else **runnable** ‚Üí append to `order`. Decrement children either way.
3. Any node never taken (in-degree never hit 0) is in a cycle ‚Üí raise `DependencyCycleError`. Deferred nodes are still taken, so a cycle is never masked.

The deferral predicate is an **exclusion** ‚Äî defer unless the blocker is `completed`/`canceled` ‚Äî so `started`/`unstarted`/`backlog`/`triage` external blockers all defer. Ready set ordered by `(sortOrder, id)`; at ‚â§15 nodes a small heap or a re-sorted list is equally fine.

## How ‚Äî orchestrator

`pending_issues` now returns `ExecutionPlan`. In `orchestrator.run` (the `RunLog` is built before the fetch): wrap the `pending_issues` call in `try/except DependencyCycleError` ‚Üí `log.set_cycle_halt(reason)`, print a `Halt:` line, `return 1`, run nothing. Before the loop, print one stderr line per `plan.deferred` ("drain-cycle: deferred ABA-NNN ‚Äî blocked by ABA-MMM (In Progress)") and skip it (no state move, no spawn, no worktree, no run-log entry). Loop over `plan.order`; `total = len(plan.order)`. If `plan.order` is empty but `plan.deferred` is not, print the stderr lines and `return 0` (expected, nothing halted). `cli.py` should need no change ‚Äî confirm the exception doesn't escape `run`.

## Behavior summary

| Situation | Result |
| -- | -- |
| No blocks | pure `sortOrder` ascending |
| A blocks B, B has lower sortOrder | A runs before B |
| Blocker Done/Cancelled | edge ignored, blocked issue runs normally |
| Blocker open & not in drain (incl. In-Progress in cycle) | blocked issue deferred (cascades) |
| Blocks cycle / self-loop among pending issues | halt, exit 1, run nothing |
