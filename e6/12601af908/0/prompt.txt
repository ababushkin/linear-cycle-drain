<title>drain-cycle: plugins (entire.io) don't load headless — capture and document the divergence</title>
<description>
## Problem

The operator noticed some normal plugins (e.g. [entire.io](<http://entire.io>)'s checkpointing) didn't take effect during the headless drain, hinting the spawn environment diverges from an interactive session more than intended. A verification spike showed hooks *do* fire under `claude -p` (entire's SessionStart hook fired when run from a normal repo root) — so the likely difference is the worker's **worktree cwd** (`.worktrees/<id>` of the target repo) and/or where entire's hook is registered. Root cause needs runtime confirmation, not a guess.

## Change (observability-first — no blind fix)

* Add an opt-in `--debug-file <path>` to the worker spawn, written next to the run log, so the next run records exactly which plugins / MCP servers / hooks initialised.
* Add a documented one-shot reproduction step to `docs/design-decisions.md`: run `claude -p` headless in a worktree vs a repo root and diff the loaded plugins/hooks.
* Add a design-decision entry naming the headless-vs-interactive divergence as a known constraint, with the open question and the spike evidence (hooks fire under `-p`; the suspect is worktree cwd).

No code change to plugin/hook loading this round.

## Acceptance criteria

* `--debug-file` capture is wired into the spawn (opt-in) and lands beside the run log.
* `docs/design-decisions.md` has the reproduction step + divergence decision entry.
* A reproduction run confirms (or refutes) the worktree-cwd hypothesis and the finding is recorded.

## Recommended worker model

**Opus** — investigation/judgment-heavy, low code.

## Files

`drain_cycle/worker.py` (opt-in `--debug-file`), `docs/design-decisions.md` (reproduction + decision), `README.md` (debug capture note).
