<issue identifier="ABA-299">
<title>Resume a halted issue on re-run (reuse the preserved worktree)</title>
<description>
## What

On re-run after a halt, drain-cycle will **reuse the preserved worktree** instead of failing on the existing branch, and spawn a **fresh** `claude` session whose prompt says "prior work is committed on this branch ‚Äî review it and continue from where it left off." A fresh session means a **clean context, so the per-issue token budget resets** (the whole reason the operator raised the cap). On Done, the existing teardown removes the worktree as normal.

**Scope.** The change is deliberately narrow: automatic only for the clean post-halt state (registered worktree + branch); bounded by a resume-attempt cap; preserves a dirty tree as-is (never resets/cleans); not in-process auto-retry. The manual escape hatch (delete worktree + branch by hand) remains the way to force a fresh start.

## Why

### The mechanics today

`drain-cycle` spawns one `claude -p` worker per Linear issue in an isolated git worktree at `<target-repo>/.worktrees/<id>` on a branch named `<id>`. When a worker is SIGKILLed mid-issue (the dominant case: a per-issue **token-cap breach**), the orchestrator reverts the issue's Linear state to its pre-run value (e.g. `Todo`, so it is re-picked next invocation) but **leaves the worktree and branch in place** for inspection (`docs/design-decisions.md` ¬ß3).

### The gap

On re-run, the issue is re-picked, but `worktree.add` runs `git worktree add -b <id> ‚Ä¶`, which fails because that branch already exists. Today this surfaces as a clean `Halt: ‚Ä¶ ‚Äî setup failed: ‚Ä¶ a branch named '<id>' already exists` (the generic pre-spawn handler already turns the `RuntimeError` into spec-shaped UX ‚Äî so the "detect cleanly" goal of the superseded <issue id="3c8d126f-d01d-4a8e-a7ac-faa69dc6027b">ABA-231</issue> is effectively met). But the run cannot proceed: the operator must manually delete the worktree/branch, **discarding any committed work the killed session produced**.

### Motivating case

A token-breached run on <issue id="f8927151-bb4d-4af7-9410-071b39c8577c">ABA-276</issue> had already committed 2 commits / 142 lines to the `ABA-276` branch before being killed. That work was the durable record of progress ‚Äî and the only way forward was to throw it away and re-drain from scratch (re-spending \~8M tokens).

### The design decision this reverses

`docs/design-decisions.md` ¬ß8 currently pins: *"Retry. A halted issue is not retried automatically. The operator inspects the worktree and decides ‚Äî fix, redo manually, or descope."* This work **partially reverses** that and so must add a new **¬ß14** that supersedes the ¬ß8 "Retry" bullet (per the "memory lives in artefacts / ADRs are superseded, not silently ignored" rule). (¬ß13 is already taken by "Opt-in OpenTelemetry tracing to Honeycomb", which landed after this issue was shaped ‚Äî the next free section number is ¬ß14; the agent should confirm the highest section at implementation time.) The reversal is narrow and must be documented as such:

* Resume reuses a **preserved, committed** work-in-progress on a **manual re-run** (or scripted re-run). It is **not** in-process auto-retry, and it is **bounded** by a resume-attempt cap (below).
* ¬ß3 (worktree preservation on halt) and ¬ß10 (`link_project_config`) are **relied upon**, not changed.
* The `docs/tasks/rerun-after-halt-detect-cleanly.md` framing ("no auto-cleanup, operator owns retry; detect-and-halt only") is superseded by this work; note it in ¬ß14. (The corresponding Linear issue, <issue id="3c8d126f-d01d-4a8e-a7ac-faa69dc6027b">ABA-231</issue>, is already Canceled/archived ‚Äî this issue replaces its stance.)

### Why not `claude --resume`

A SIGKILLed worker has `session_id: null` in the run log (it dies before the terminal `result` event), so `--resume` isn't even available. And `--resume` would reload the exact context that blew the token budget, re-breaching almost immediately. The committed branch ‚Äî not the session ‚Äî is the correct resume point.

### Assumptions (the core bet)

* **A1.** The dominant halt is a token-cap breach, and a killed worker has usually committed incremental work to the `<id>` branch (true in the <issue id="f8927151-bb4d-4af7-9410-071b39c8577c">ABA-276</issue> case: 2 commits). If a worker dies before committing anything, resume re-runs from an empty branch ‚Äî no worse than today, and bounded by the cap (A3).
* **A2.** Only the clean post-halt state (registered worktree + branch) is worth auto-handling. Operator-induced partial states (branch without worktree, orphan dir) are rare and acceptably handled by the **existing** clean halt ‚Äî no new code. This is the main simplification; if partial-state auto-recovery turns out to be wanted, it's an additive follow-up.
* **A3.** A fresh session, pointed at committed work and told to continue, will build on it rather than restart. This is the core bet; the resume prompt directive (`git log`/`git status` first) is the cheap insurance, and the resume-attempt cap is the backstop if the bet fails for a given issue.

## How

### Resolved design choices

* **Resume is automatic on re-run** (the operator's explicit intent). Forcing a fresh start is the manual escape hatch: delete the worktree + branch (`git worktree remove ‚Ä¶ && git branch -D <id>`), exactly as we just did for <issue id="f8927151-bb4d-4af7-9410-071b39c8577c">ABA-276</issue>. No new flag needed.
* **Resume-attempt cap = 3** (configurable). After 3 non-Done attempts of an issue within the cycle, drain-cycle halts **before spawning** (zero token burn) and leaves the issue for manual intervention. This is the only backstop: the cycle-wide token cap does **not** fire on repeated halts ‚Äî the breach path `return 1`s before `check_cycle`, and each re-run is a new process with a fresh `RunLog`, so cumulative cycle tokens reset every invocation.
* **Dirty preserved tree: preserve as-is.** The fresh session reconciles uncommitted edits via `git status`/`git log`. Never `reset --hard`/`clean` (would destroy the saved work and the gitignored config symlinks). Never switch teardown to `--force`.

### Worktree reuse logic (two branches, not five)

For picked issue `<id>` in `target_repo`, with `path = target_repo/.worktrees/<id>`:

1. **A worktree is already registered at** `path` ‚Üí reuse it. Run **no git command** ‚Äî just return `(path, resumed=True)`. This is the clean post-halt state (the <issue id="f8927151-bb4d-4af7-9410-071b39c8577c">ABA-276</issue> case) and the only state worth auto-handling.
2. **Otherwise** ‚Üí call today's `add` (`git worktree add -b <id> <path> main`), returning `(path, resumed=False)`.

That's it. The rare leftover states ‚Äî branch exists but no worktree, or an orphan directory at `path` ‚Äî make `add` raise `RuntimeError` (with git's stderr), which the orchestrator's **existing** pre-spawn handler already turns into the clean `Halt: ‚Ä¶ ‚Äî setup failed: ‚Ä¶` line. We do **not** add bespoke detection, `prune`, branch re-checkout, or a new exception type for those: they're operator-induced, rare, and the operator clears the leftover by hand (`git worktree remove ‚Ä¶ && git branch -D <id>`). **The detection is one git call** (`git worktree list --porcelain`, parsed for a record at `path.resolve()`) plus the existing `add`.

Verified against real git: any `git worktree add` onto an already-registered path fails, so the reuse case must **not** call `add` ‚Äî hence the registration check first. Reuse is a pure no-op on git state, so a dirty index / staged / untracked files in the preserved worktree survive untouched (this is the "preserve as-is" choice, for free).

### Component changes

#### 1\. `drain_cycle/worktree.py` ‚Äî new `ensure` (keep `add` as-is)

```python
@dataclass(frozen=True)
class WorktreeHandle:
    path: Path
    resumed: bool

def ensure(repo: Path, identifier: str) -> WorktreeHandle:
    """Reuse a preserved worktree if one is registered at the expected
    path, else create a fresh one via add(). Reuse runs no git command."""
    path = repo / WORKTREE_DIR / identifier
    if _is_registered_worktree(repo, path):
        return WorktreeHandle(path, resumed=True)
    return WorktreeHandle(add(repo, identifier), resumed=False)
```

* One private helper: `_is_registered_worktree(repo, path) -> bool` ‚Äî runs `git worktree list --porcelain` (via the existing `_run_git`-style call) and checks for a `worktree` record equal to `path.resolve()`. No `WorktreeStateError`, no `prune`, no branch re-checkout, no `_branch_exists`.
* **Keep** `add(repo, identifier) -> Path` **exactly as-is** ‚Äî `tests/test_worktree.py::test_add_failure_raises_with_git_stderr_in_message` pins it as the fail-on-duplicate primitive that `ensure` reuses for the fresh/leftover path, and its `RuntimeError` is what the orchestrator's existing handler turns into the clean halt.
* `link_project_config` is already idempotent (`os.path.lexists` skip); call it unconditionally after `ensure` for both fresh and resumed (confirmed safe ‚Äî returns `[]` on reuse).

#### 2\. `drain_cycle/prompt.py` ‚Äî `resumed` kwarg

```python
def build(issue: dict[str, Any], worktree: Path, *, resumed: bool = False) -> str: ...
```

* When `resumed=True`, prepend a resume segment as the **first line of the preamble** (after the `---` separator, before "Execution instructions:") so it leads the instructions but **does not displace** `_TAIL` **from last position** (the four-segment ordering is load-bearing). Text directs the agent to run `git log --oneline main..HEAD` and `git status` first and continue rather than restart.
* `resumed=False` default keeps every current call site and all existing `tests/test_prompt.py` assertions unchanged.

#### 3\. `drain_cycle/limits.py` ‚Äî `max_resume_attempts`

* Add `max_resume_attempts: int | None = 3` to the `Limits` dataclass. It is a **count**, not a resource ceiling, but lives here because `limits.yml` is the natural config home; `null` disables. The existing `_coerce`/`load` path validates it (positive number or null; rejects bool/non-positive) with no change.

#### 4\. `drain_cycle/orchestrator.py` ‚Äî swap `add`‚Üí`ensure`, thread flag, add cap guard

In the pre-spawn try-block (`orchestrator.py:169-205`), replace `worktree.add(...)` with `worktree.ensure(...)` and insert the cap check **before** `linear.set_state(..., "In Progress")` (no spawn ‚áí no Linear transition, mirroring the resolution-halt contract):

```python
handle = worktree.ensure(target_repo, identifier)   # RuntimeError on leftover/conflict ‚Üí existing halt
worktree_path = handle.path
worktree.link_project_config(target_repo, worktree_path, repos.worktree_config_paths)
if handle.resumed and limits.max_resume_attempts is not None \
        and _resume_attempts(log.cycle_id, identifier) >= limits.max_resume_attempts:
    # clean halt BEFORE set_state: run-log entry (cached issue state), stderr Halt line, return 1.
    # No revert (state was never moved). Distinct reason names the cap.
    ...
linear.set_state(issue["id"], _IN_PROGRESS_STATE_NAME)
...
agent_prompt = prompt.build(issue, worktree_path, resumed=handle.resumed)
```

* New `_resume_attempts(cycle_id, identifier) -> int`: glob `runlog.runs_dir()/<cycle_id>-*.json` (reusing the per-cycle filename convention) and count entries with `issue_identifier == identifier` and `final_linear_state != "Done"` (prior non-Done attempts). The current run's just-created empty file contributes nothing.
* The `resumed` flag flows **only** to `prompt.build`, never to `worker.run_issue` ‚Äî the worker always spawns a fresh session (`session_id` is null on kill; `--resume` is unavailable and undesirable).
* Cap-halt reuses the `_halt_message` helper with an appended reason, so the stderr line and the run-log `halt_reason` stay single-sourced.

#### 5\. Docs

* `docs/design-decisions.md`: add **¬ß14** (resume on re-run) documenting the reversal of the ¬ß8 Retry bullet, the state matrix, the resume cap, the dirty-tree choice, and why `--resume` is not used. Annotate the ¬ß8 Retry bullet as superseded by ¬ß14. (Confirm the next free section number at implementation time ‚Äî ¬ß13 is OpenTelemetry tracing.)
* `README.md`: the halt example currently says the worktree is "left in place"; add that a re-run now **resumes** it, and document `max_resume_attempts` alongside the other `limits.yml` keys.
* `docs/tasks/rerun-after-halt-detect-cleanly.md`: note it is superseded by this feature.

`tests/test_worktree.py` (real git, existing `_init_repo`):

* fresh ‚Üí `WorktreeHandle(resumed=False)`, worktree+branch created.

`tests/test_orchestrator_resume.py` (new; mirror `test_orchestrator_iteration.py`):

`tests/test_prompt.py`: `resumed=True` includes the resume directive + identifier and `_TAIL` is still the last non-empty line; `resumed=False` is byte-identical to today's prompt.

`tests/test_limits.py`: `max_resume_attempts` honours absent-key default (3), explicit `null` (off), positive override, and rejects bool/non-positive via `_coerce`.

## Delivery ‚Äî author as a 4-PR Graphite stack

**This issue is not drained via drain-cycle and is not split into Linear sub-issues.** It is authored by **one supervised agent in a single session**, working on a branch off `main` in the repo root (not a `.worktrees/` sandbox), and delivered as a **Graphite (**`gt`**) PR stack** ‚Äî one PR per layer, reviewed bottom‚Üítop. This is the owner-authorised exception to the AGENTS.md "push directly to main; PRs only when the owner asks" default. <issue id="21915a3a-c443-4c9e-a032-b0964f0ed845">ABA-299</issue> stays the single tracked issue.

**Precondition:** `gt` (1.8.x) authenticated against `origin` (`github.com/ababushkin/drain-cycle`); the session is supervised, so branch/PR creation runs with approvals. The repo has **no CI**, so each layer's gate is a local `pytest` run ‚Äî not a remote check.

**The stack (maps the 7 tasks above onto 4 PRs):**

| PR | Layer | Tasks | Contents | Green on its own |
| -- | -- | -- | -- | -- |
| PR1 | worktree primitive | 1, 2 | `WorktreeHandle`, `_is_registered_worktree`, `ensure`; `test_worktree.py` reuse/leftover tests. `add` untouched. | yes (its unit tests) |
| PR2 | prompt variant | 3 | `prompt.build(resumed=)` + resume segment; `test_prompt.py`. | yes |
| PR3 | limits knob | 4 | `max_resume_attempts` on `Limits`; `test_limits.py`. | yes |
| PR4 | wiring + docs | 5, 6, 7 | orchestrator `add`‚Üí`ensure`, thread `resumed`, `_resume_attempts` + cap guard; `test_orchestrator_resume.py`; **and docs (¬ß14, README, supersession note) in this same PR**. | yes (full suite) |

PR1‚Äì3 are independent leaves (different files), each green on its own even though no production caller uses them yet. PR4 is the only behaviour-changing layer and integrates the three below it. **Docs land in PR4** so the ¬ß8-reversal ADR ships in the same PR as the behaviour that reverses it (an architectural change updates its ADR in the same PR ‚Äî don't trail the docs in a later PR).

**Authoring loop ‚Äî one layer at a time, gated:**

1. `gt create` a branch for the layer (conventional-commit subject, e.g. `feat: worktree.ensure reuses a preserved worktree`).
2. Run `pytest` ‚Äî **the layer's own tests must be green** before submitting (no CI; local pytest is the only gate).
3. `gt submit` ‚Äî opens the real GitHub PR for that layer on `origin`.
4. **STOP and report the PR; wait for the operator's go-ahead.** Do **not** start the next layer until approved.
5. On approval, build the next layer **on top** with `gt create` (never branch from `main` for layers 2‚Äì4 ‚Äî the stack is one lineage; `gt` sets the parent).
6. On review feedback for a lower layer: `gt down`, amend, `gt restack`, `gt submit` to update the stack. The same agent restacks the whole tower ‚Äî it has full context, which is the reason one agent authors all four layers rather than handing each to a fresh worker.

**Completion ritual is adapted for the stack:** `/code-review-and-quality` + commit happen **per layer** (before each `gt submit`); the Linear issue moves to **Done once the whole stack is approved and merged**, not per layer. Do not push individual layers straight to `main`.

## Acceptance criteria

Observable end-to-end checks, verified after the stack merges. (`pytest` green is the per-PR-layer precondition ‚Äî see Test plan ‚Äî not a behavioural criterion.)

* **Headline resume.** In a throwaway repo, `worktree.add` an issue branch, commit a file, leave the worktree; run the orchestrator (with a fake/cheap model) against a stubbed cycle containing that issue ‚Üí it resumes (prompt mentions prior work), completes, and the worktree is removed on Done.
* **Token budget resets.** The resumed run's log entry has a non-null, distinct `session_id` and `cumulative_tokens` reflecting only this session's spend ‚Äî not the killed run's.
* **Manual escape hatch.** After `git worktree remove ‚Ä¶ && git branch -D <id>`, the next drain creates a fresh worktree (`resumed=False`) and runs normally.
* **Dirty preserved tree.** With staged + untracked edits left in the preserved worktree, drain runs and the resumed session sees them via `git status` ‚Äî no `reset`/`clean` ran; the symlinked config dirs survive.
* **Leftover partial state.** With the worktree directory deleted but the branch kept, drain halts cleanly with `Halt: ‚Ä¶ ‚Äî setup failed: ‚Ä¶` (no traceback, no Linear transition).
* **Resume cap fires.** Seed 3 non-Done run-log entries for an id; the next run halts before spawn with the cap reason, `set_state` not called.
* **Cap disable.** With `max_resume_attempts: null` in `limits.yml` and ‚â•3 seeded non-Done entries, drain proceeds rather than halting.

## Out of scope

(Deliberate, v2 candidates.)

* **Auto-recovery of operator-induced partial states** (branch without worktree ‚Üí re-checkout; orphan/prunable dir ‚Üí prune). Deliberately cut for simplicity (A2): these hit the existing clean halt and the operator clears the leftover by hand. Add only if it proves needed.
* Per-resume token budget (a tighter cap for resumed sessions) ‚Äî the count cap is the load-bearing guard for v1.
* Skipping a capped issue to drain the rest of the cycle (today any halt stops the run; a capped issue blocks the cycle until the operator intervenes ‚Äî consistent with the existing halt contract).
* A `drain-cycle clean <id>` recovery subcommand.
* `claude --resume` session continuation (unavailable on kill; undesirable for token-breach).
